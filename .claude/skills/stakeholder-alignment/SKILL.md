# Stakeholder Alignment

Consolidate the stakeholder signal that other skills have already scattered across the
hub (`evidence-synthesizer` logs stances, `distribution-planner` surfaces channel
dependencies) into one clear picture: who can move this forward or stop it, and what
they each need to believe to do the former instead of the latter.

This skill prepares the human for stakeholder work — it doesn't replace their judgment,
visibility, or political read of the room. Say so if a request implies otherwise.

## Before you start

Read `11-stakeholders.md` for whatever's already there from earlier passes — don't
restart from scratch or duplicate rows that already exist. Skim `07-positioning.md` /
`08-messaging-house.md` / `10-distribution.md` for what's actually being proposed, since
"what does this stakeholder need to believe" only makes sense relative to something
concrete.

## Process

1. **Consolidate, don't duplicate.** Pull together stakeholder rows already logged
   elsewhere in the hub, fill gaps in the standard roles (Product Management, Sales,
   Marketing, Support, Brand, Leadership) if evidence exists for their stance, and add
   any missing stakeholder the current proposal actually depends on.
2. For each stakeholder, be concrete about **decision power** — can they approve, block,
   dilute (slow-walk, water down), or accelerate this? A stakeholder with no real power
   over the outcome doesn't need a *new* row; but don't delete a row another skill
   already logged just because this pass finds their power marginal — annotate it
   instead, since a prior pass may have seen evidence you haven't. If a standard role
   (Product, Sales, Marketing, Support, Brand, Leadership) has no evidence of a stance
   at all, say so explicitly in its row rather than leaving it blank or inventing one.
3. Translate the current strategy into **what each stakeholder specifically needs to
   believe** to support it — not a generic "buy-in," the actual claim or reassurance that
   would move them.
4. **Separate unresolved decisions from execution tasks.** "Which pricing number is
   correct" is a decision someone must make. "Send the Slack message asking" is a task.
   Don't let a real decision hide inside a to-do list disguised as a next step.
5. Write Alignment Risks for anything that could block, dilute, or derail — with why it
   matters and a mitigation, not just a description of the risk.

## Rules

- Don't automate away the human part. This skill prepares the brief; it doesn't run the
  meeting, read the room, or make the political call.
- Every stakeholder row needs real decision power behind it — this isn't a full org
  chart, it's the people who can actually change the outcome.
- Keep unresolved decisions and execution tasks in visibly separate lists.

## Output

Updates: `11-stakeholders.md` (Stakeholders, Alignment Risks, Unresolved Decisions),
`00-index.md` (Freshness row, Next Best Action).

End with the single next best action — usually naming the single highest-leverage
unresolved decision to force first, or `campaign-starter` if stakeholder alignment is the
last gate before launch assets.
